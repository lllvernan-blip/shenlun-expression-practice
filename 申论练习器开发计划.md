## 二、项目目标

当前项目已有两个训练模式：概括训练（选点、命名、成句）和规范改写（口语句改写为正式表达）。本轮优化只聚焦概括训练，规范改写模式保留，不扩展新功能边界。

优化后的核心闭环：

```text
阅读完整材料
→ 识别有效事实
→ 排除背景、例子和干扰信息
→ 将多个事实合并为共同含义
→ 写出上位概括词
→ 组织完整答案
→ 根据证据逐层反馈
```

成功标准不是"分数显示出来"，而是使用者做完一道题后能明确知道：我漏掉了哪条材料事实；哪几条事实应该合并；我写的概括词为什么不够准确；最终答案缺了什么，或者为什么表达不完整。

## 三、离线与 AI 功能边界

这两条线必须分开设计、分开验收，不能把离线规则反馈包装成 AI 批改。

### 离线模式

离线模式不调用 API，不需要 API Key，也不承担开放式语义判断。它只基于题库中预先编写的结构化数据进行稳定、可复查的反馈：根据 slice 的 `valid`/`point` 判断事实是否漏选或误选；根据 `points` 判断哪些核心含义被覆盖；根据 `acceptable` 判断概括词是否命中预设的可接受表达；根据 `buildChain` 展示参考答案如何从事实搭出来。

离线反馈的措辞要诚实，例如"按本题预设答案判断""系统识别到你覆盖了 2/3 组含义"，不要写成"AI 认为你的答案……"或暗示系统具备开放式语义理解。

### AI 模式

AI 模式单独处理需要模型判断的功能：AI 水平测试、根据学习者画像生成新题、AI 对答案进行开放式批改、AI 扫描文本并提出规范表达建议。

AI 模式必须明确显示联网/API 状态、模型调用状态和失败降级结果。AI 批改失败时可以退回离线规则反馈，但要标明"以下为题库规则反馈，不是 AI 批改"。不要让离线规则和 AI 反馈混在同一个结论里。

### 共同的界面原则

概括训练的题目和答题流程可以共用，但提交后的反馈要标记来源：`离线规则反馈`基于本题预设映射，稳定可复查；`AI 批改反馈`由当前配置的模型生成，可能存在不确定性。

离线模式必须在没有网络、没有 API Key、直接打开单文件时正常工作。AI 模式不能成为离线核心训练闭环的前置条件。

## 四、题库时效与现实依据

离线题库不是随便编几段"像申论"的材料。材料应尽量来自真实、可核查的政策实践、政府工作报告、权威媒体报道或公开调研材料，再进行必要的压缩和改写。可以保留材料的事实结构，不必整段复制原文。

### 时效要求

每道题带 `source`、`sourceDate`、`sourceType`、`topicTags`、`timeliness`、`timelinessNote` 元数据。`sourceDate` 使用明确年月日，无法确认日期时不要伪造。涉及政策状态、阶段性目标、统计数据和工程进展时，必须检查截至当前制作日期是否仍然成立；不使用已经发生明显变化的表述作为当前事实。对"脱贫攻坚""全面建成小康社会"等已经完成的阶段性历史议题，只能作为历史背景、经验总结或政策演进材料，不能写成"当前正在脱贫"。

```js
source: "来源名称或公开链接",
sourceDate: "2026-07-01",
sourceType: "政府公开材料/权威媒体/政策文件/案例改写",
topicTags: ["基层治理", "公共服务"],
timeliness: "current",
timelinessNote: "基于公开案例压缩改写，政策表述按制作日期核对。"
```

如果是原创综合材料，必须明确标记为 `sourceType: "原创综合材料"`，不能伪装成真实报道。原创题可以用于训练，但涉及现实政策、统计数据和时间状态的内容必须可核查。

## 五、API 配置持久化与恢复

API 配置属于独立的本地设置，不应因为清空练习记录、切换离线模式、重建题库或数据文件版本变化而丢失。

主应用已把 API 配置从练习数据中拆成独立存储 `API_SETTINGS_KEY='shenlun_api_settings_v1'`（字段 `provider`/`apiKey`/`model`/`baseUrl`/`temperature`/`maxTokens`），`loadApiSettings`/`saveApiSettings` 读写，`mergeApiSettingsFromFile` 本地优先合并，`migrateApiSettingsFromLegacy` 一次性迁移老 `D.settings.apiKey`。样板已与主应用共用此键并对齐 PROVIDERS。

目标行为：填写并保存 API Key 后，关闭重开、切离线、清空练习记录、连接旧数据文件，配置都不丢；旧数据文件的空配置不能覆盖本地已保存配置；API 请求失败不能自动清空输入框或覆盖已保存配置；不把真实 API Key 写进仓库、题库、Git 历史或日志。

待验收的持久化场景：保存后刷新；关闭浏览器后重开；先连接旧数据文件再打开设置；切换离线再切回在线；清空练习历史；仅修改模型或 Base URL 后重开；API 请求失败后重开；localStorage 无配置但旧数据文件有配置；两边冲突时界面明确采用哪一份，不静默覆盖。

## 六、手机端使用与本地保存

手机端是实际使用场景，尤其要考虑上课期间用手机做几道题。目标不是让手机连接电脑上的文件，而是让手机浏览器能独立打开、练习、保存和恢复数据。

### 手机端目标

Android 和 iPhone 浏览器都能打开练习器；不依赖桌面端 `.bat`、Python 服务或本地开发环境；手机窄屏下材料、事实选项、命名输入框和答案区不发生横向溢出；触摸选择、输入法、滚动和提交按钮正常工作；离线题库在没有网络时仍可使用；练习记录自动保存在当前手机浏览器；关闭页面后重新打开，题目记录、历史和设置仍能恢复。

### 本地保存边界

`localStorage` 和 IndexedDB 是手机浏览器内的本地保存，不是跨设备同步，也不是用户可见的文件，清理浏览器数据后可能丢失。桌面端的 File System Access API 不能作为手机端唯一保存方案（iPhone 尤其不能持续写入用户选择的本地文件）。因此手机端必须先保证浏览器本地存储独立可用。

### 必须增加的数据备份能力

为避免"手机清缓存后全没了"，在设置或记录页增加：`导出数据`（练习记录、错题、画像、设置导出为 JSON）、`导入数据`（从文件选择器恢复 JSON）、`复制数据`（复制为文本的兜底方式）；导入前显示数据时间和记录数量并要求确认；导入失败不能覆盖现有数据；API Key 默认不导出，或单独提供明确的"是否包含 API 配置"选项并警告。

### 本轮部署决策：先做单文件下载版

本轮明确不部署 HTTPS，也不做 GitHub Pages、Cloudflare Pages、Netlify 或国内云平台发布。先把 `规范表达练习.html` 做成可以直接分享和下载的单文件版本：电脑下载后双击打开；手机下载后用浏览器或文件管理器打开；GitHub 私有仓库只作代码备份；练习数据保存在当前设备浏览器本地，并通过导出/导入 JSON 备份。未来如果决定公开给别人使用，再单独讨论国内可稳定访问的 HTTPS 方案。

## 七、评价方式：离线对照，AI 逐语义单元批改

### 1. 先区分"逐字逐句"

不建议按每个汉字机械扣分。申论答案允许同义替换、语序调整和不同但合理的规范表达，逐字比较会把正确答案误判为错误。本计划中的"逐字逐句评价"指：按句子或语义单元拆分用户答案；每个语义单元对应一个材料事实、采分点或答案结构；标出原文片段、对应的标准要点、判断结果和修改建议；对规范词进行词语级提示，但不把单个字的差异直接当成扣分依据。

### 2. 离线模式：取消笼统评分，改为答案对照

离线模式不做一个看似精确但不可靠的总分，也不对用户自由答案做个性化语义判断。概括训练提交后，展示"你的答案 / 参考拆解 / 参考答案"的对照；离线规则只核对找点、干扰项和归类命名。不要显示覆盖率、答案长度、数字使用等针对个人答案的诊断，也不要显示"你得了 7.3 分"这类精确结论。离线反馈的数据来源必须标注："离线答案对照 · 题库预设 · 非 AI 批改。"

### 3. AI 模式：逐语义单元、逐句解释

AI 模式可以进行更开放的评价，但必须要求模型返回结构化结果，不接受一段笼统总评代替逐句批改。建议返回结构含 `overall`、`units[]`（每个含 userText/matchedPoint/status/problemType/comment/suggestion/referenceText/score，status 为 covered/partial/missing/extra/uncertain）、`dimensions`（要点覆盖/概括准确/表达规范/结构完整）、`nextFocus`。如果模型无法可靠判断某个表达是否等价，应标记为 `uncertain`，提示用户对照，而不是直接扣分。

### 4. 逐句反馈的界面要求

离线模式显示题库预设的确定性对照；AI 模式显示模型生成的逐句反馈，并在顶部明确标注来源。两者不能共用一个没有来源说明的"综合评分"。

## 八、当前主要问题

### 1. 材料与训练目标不匹配

旧题目材料偏短，且使用者更像是在选择预先拆好的句子，而不是从完整材料中主动找事实。新流程用 slice 模型让材料完整连贯、干扰项内联，解决这一问题。本轮先把材料控制在约 150 至 350 个汉字，作为微型材料训练；后续如果机制验证有效，再逐步增加至 400 至 800 字或更长。

### 2. 评分逻辑过度依赖字符串包含

离线规则可以用字符串匹配辅助定位，但不能把字符串命中等同于真实评分。AI 模式需要单独进行语义单元匹配，并把不确定判断显式标记出来。

### 3. 解析告诉了结果，没有解释过程

反馈要从"判分"改成"指出下一次怎么做"：漏掉哪条事实、误选了哪条事实、哪些事实构成同一组、当前答案和参考归类之间的差异、为什么某个概括词比口语词更合适。

## 九、题库数据结构：slice 模型

SUMMARY_BANK 统一采用样板的 slice 数据模型。核心是把连续材料切成可点选的片段，干扰项内联在材料里，每片标注是否有效、归属哪个采分点。

```js
{
  id: "s01",
  type: "做法",          // 题型：问题/做法/成效/变化/经验
  level: "L2",
  domain: "社会治理",
  context: "常德市武陵区2025年民生工作",

  // 来源元数据（见第四节）
  source: { name, url, fetchedAt, authenticity },
  sourceDate: "2025-01-01",
  sourceType: "政府公开材料(材料为压缩改写)",
  topicTags: ["基层治理", "公共服务"],
  timeliness: "current",
  timelinessNote: "……",

  // 完整连续材料（= 所有 slice 的 text+punct 顺次拼接）
  material: "我们以持久之功兴利民之事。坚持将75%以上的财力写入民生账本，……",

  // 切片：text+punct 严丝合缝拼回原文
  slices: [
    { text: "我们以持久之功兴利民之事", punct: "。", valid: false, reason: "修辞性总起，不是具体事实" },
    { text: "坚持将75%以上的财力写入民生账本", punct: "，", valid: true, point: 0 },
    { text: "省市37件重点民生实事全面完成", punct: "，", valid: false, reason: "总括性背景表述，不是具体要点" }
    // ……
  ],

  // 采分点：name 上位概括词，keys 判断线索词，acceptable 可接受同义表达
  points: [
    { name: "民生投入力度持续加大", keys: ["财力","民生账本"], acceptable: ["加大民生投入","财力向民生倾斜"] }
    // ……
  ],

  // 参考答案搭建链：事实 → 共同含义
  buildChain: [
    { facts: ["将75%以上财力写入民生账本"], cat: "民生投入力度持续加大" }
    // ……
  ],

  // 参考答案
  ref: "武陵区持续加大民生投入力度，不断优化教育、养老、托育等公共服务供给，……"
}
```

### 数据要求

每个核心 point 必须至少对应 1 条事实，重要 point 最好对应 2 条相互支撑的事实。不要让 `name` 直接复述材料原词——材料写"跑几个窗口"，概括词应当是"办事流程繁琐"或"服务流程不够优化"，而不是"跑窗口问题"。每道材料至少加入 1 条不应被概括为核心要点的背景或例子（干扰项），并写清 `reason`。

现有 4 道题（s01–s04，均为"做法"型）需重写为 slice 格式：把 `material` 切成 slice，标出 valid/point，补干扰项与 reason，补 buildChain。重写时保留原材料事实结构，不虚构新来源；若原题无明确来源，标 `sourceType: "原创综合材料"`。

## 十、交互方案：一步一屏

概括训练改为分步向导，每一步只聚焦一个动作，且每步都带出当前动作需要的前文，不让用户凭记忆回想材料。

### 第 1 步 · 读材料

通读完整连续材料，显示来源名称、原文链接、抓取日期和真实性说明。

### 第 2 步 · 找点

材料以可点选片段（slice）呈现，点击高亮支撑答案的有效信息，再点取消。引导语："圈出能支撑答案的事实，注意排除背景、举例和修辞性描述。"提交后明确区分漏选事实、误选背景或干扰项（显示 reason）、选中了事实但没有覆盖对应 point。

### 第 3 步 · 归类 · 命名

先填写采分点数量，再把已选事实分配到要点组，为每组写一个上位概括词。可回看材料原文。命名匹配至少支持：完全匹配、去除"问题/现象/不足/程度"等尾词后匹配、预定义可接受同义表达（`acceptable`）、无法判断时标记"待对照"而非直接判错。

### 第 4 步 · 成句

照着整理出的要点和概括词连成完整答案，可展开材料回看，显示要点名称和事实积木。保留答案输入框和字数统计。

### 第 5 步 · 对照结果

见第十一节反馈模块。

## 十一、反馈模块

每层都要包含"结果 + 证据 + 下一步建议"。

### 1. 找点反馈

显示核心事实覆盖数、漏选事实列表、误选事实列表；干扰项说明为什么不应纳入答案。误选/漏选明细超过 3 条时折叠（"展开全部（还有 N 条）"↔"收起"）。

### 2. 归类 · 命名反馈

逐个采分点评价：准确 / 停留事实层 / 待对照 / 未命名，并显示参考要点与用户概括的对照。

### 3. 参考答案搭建链

展示"材料事实 → 共同含义 → 组成完整答案"的搭建过程，让用户看到答案是怎么从事实搭出来的。

### 4. 答案逐点分析

答案覆盖芯片（每个采分点 ✓/✗）、用户答案命中高亮、逐点诊断（漏了哪个点、对应哪组事实没概括进去；是否罗列材料数字而非上位概括；字数是否偏长）。诊断超过 3 条时折叠。

离线模式顶部标注"离线 · 直接对答案 · 题库预设 · 非 AI 批改"；AI 模式在离线对照基础上增加 AI 个性化点评块，顶部标注"AI 批改 · 模型:XXX"，失败时降级并标"AI 失败降级"。

## 十二、评分规则

离线模式不输出正式总分，只输出答案对照和过程统计（如"覆盖 2/3 组采分点"），这个统计不是考试得分。AI 模式可以保留 10 分制，但分数只作为逐句批改后的摘要，不得替代逐句反馈；AI 分数必须来自独立的维度评分和结构化 `units`，不能复用离线结果冒充 AI 评分。

避免以下做法：只检查答案是否包含参考词；用参考词前两字、后两字直接判定正确；用户写了合理同义词却直接判错；把"显示参考答案"当成解析完成。如果规则无法可靠判断，显示"部分匹配"或"需要对照判断"，不要伪装成精确评分。

## 十四、工作边界与产品口径

### 执行边界

- 只改当前项目目录里的单文件 HTML 和必要的备份；改之前在 `_backups/` 保存版本。
- 不新增框架、不引入构建工具、不拆分外部依赖；不更换单文件 HTML 架构。
- 不重做整体视觉，沿用"宣纸朱批"体系。
- 不现场让 AI 编造材料、来源或链接。
- 离线反馈只能用题库映射和确定性规则，不能偷偷调用 AI；离线概括训练不输出伪精确总分。
- AI 反馈必须标注来源，不能把离线规则结果写成 AI 结论；AI 批改必须返回逐语义单元结果。
- 不部署、推送、提交、公开发布；不改动 File System Access API、localStorage、IndexedDB 等已有持久化机制的核心逻辑。
- 如果发现现有代码与本计划不一致，以实际文件和实际运行结果为准，并在继续修改前说明差异。

### 产品口径

- 题库材料统一来自预抓取并核验的真实文章；AI 不现场编材料，只负责挑题、判分和个性化分析。
- 离线模式与 AI 模式必须在界面和反馈中明确区分。
- 长反馈不能默认全部铺开，真实错很多时也要能收起。
- 一步一屏每一步必须带出当前动作需要的前文，不能让用户凭记忆回想材料。
- 目标设备包括手机，窄屏检查是必要验收项。

## 十五、最终判断标准

这轮优化成功，不是因为页面更复杂，也不是因为反馈卡片更多，而是因为离线模式和 AI 模式各自做清楚了自己的事，并且离线模式本身已经能让使用者完成一次真实的认知练习：

> 从一段材料中找出分散事实，判断哪些事实属于同一个问题或做法，给这组事实命名，再把它组织成申论答案。

如果使用者做完后只是看到了一个分数和一份参考答案，说明这轮没有完成。

## 工作方式要求

阿楠对"已完成、已修复、已验证"非常敏感。报告状态时必须区分：已写入代码 / 已通过语法检查 / 已浏览器实测 / 尚未实测或仅推测。
