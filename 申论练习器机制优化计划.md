# 申论表达练习器机制优化计划

> 交给其他 Agent 执行。目标不是重做视觉，而是把当前工具从“规范词小游戏”优化成一套真实训练“材料事实 → 共同含义 → 上位概括 → 规范答案”的练习工具。

## 一、项目目标

当前项目已有两个训练模式：

- 概括训练：选点、命名、成句
- 规范改写：口语句改写为正式表达

本次优化只聚焦概括训练。规范改写模式保留，不要在本轮扩展新的功能边界。

优化后的核心闭环应当是：

```text
阅读完整材料
→ 识别有效事实
→ 排除背景、例子和干扰信息
→ 将多个事实合并为共同含义
→ 写出上位概括词
→ 组织完整答案
→ 根据证据逐层反馈
```

成功标准不是“分数显示出来”，而是使用者做完一道题后能明确知道：

1. 我漏掉了哪条材料事实；
2. 哪几条事实应该合并；
3. 我写的概括词为什么不够准确；
4. 最终答案缺了什么，或者为什么表达不完整。

## 二、离线与 AI 功能边界

这两条线必须分开设计、分开验收，不能把离线规则反馈包装成 AI 批改。

### 离线模式

离线模式不调用 API，不需要 API Key，也不承担开放式语义判断。它只基于题库中预先编写的结构化数据进行稳定、可复查的反馈：

- 根据 `sentenceIndexes` 判断事实是否漏选或误选；
- 根据 `points` 判断哪些核心含义被覆盖；
- 根据 `acceptableNames` 判断概括词是否命中预设的可接受表达；
- 根据 `explanation` 展示为什么这个事实对应这个上位词；
- 根据 `answerStructure` 检查总括对象、要点覆盖和收束结构。

离线反馈的措辞要诚实，例如“按本题预设答案判断”“系统识别到你覆盖了 2/3 组含义”，不要写成“AI 认为你的答案……”或暗示系统具备开放式语义理解。

### AI 模式

AI 模式单独处理需要模型判断的功能：

- AI 水平测试；
- 根据学习者画像生成新题；
- AI 对答案进行开放式批改；
- AI 扫描文本并提出规范表达建议。

AI 模式必须明确显示联网/API 状态、模型调用状态和失败降级结果。AI 批改失败时可以退回离线规则反馈，但要标明“以下为题库规则反馈，不是 AI 批改”。不要让离线规则和 AI 反馈混在同一个结论里。

### 共同的界面原则

概括训练的题目和答题流程可以共用，但提交后的反馈要标记来源：

- `离线规则反馈`：基于本题预设映射，稳定可复查；
- `AI 批改反馈`：由当前配置的模型生成，可能存在不确定性。

离线模式必须在没有网络、没有 API Key、直接打开单文件时正常工作。AI 模式不能成为离线核心训练闭环的前置条件。

## 三、题库时效与现实依据

离线题库不是随便编几段“像申论”的材料。材料应尽量来自真实、可核查的政策实践、政府工作报告、权威媒体报道或公开调研材料，再进行必要的压缩和改写。可以保留材料的事实结构，不必整段复制原文。

### 时效要求

- 每道题增加 `source`、`sourceDate`、`sourceType`、`topicTags` 等元数据；
- `sourceDate` 使用明确年月日，无法确认日期时不要伪造；
- 涉及政策状态、阶段性目标、统计数据和工程进展时，必须检查截至当前制作日期是否仍然成立；
- 不使用已经发生明显变化的表述作为当前事实；
- 对“脱贫攻坚”“全面建成小康社会”等已经完成的阶段性历史议题，只能作为历史背景、经验总结或政策演进材料，不能写成“当前正在脱贫”；
- 涉及长期有效的治理问题，也要避免使用过时的时间、数字和政策口径；
- 每道题保留一条简短的“时效备注”，说明这是当前事实、历史案例、政策经验还是经改写的公开材料。

### 题库审核记录

建议在题目对象中增加：

```js
source: "来源名称或公开链接",
sourceDate: "2026-07-01",
sourceType: "政府公开材料/权威媒体/政策文件/案例改写",
topicTags: ["基层治理", "公共服务"],
timeliness: "current",
timelinessNote: "基于公开案例压缩改写，政策表述按制作日期核对。"
```

如果是原创综合材料，必须明确标记为 `sourceType: "原创综合材料"`，不能伪装成真实报道。原创题可以用于训练，但涉及现实政策、统计数据和时间状态的内容必须可核查。

本轮至少审核 6 道样板题，确认：事实来源不虚构、时间状态不冲突、政策口径不过时、历史案例没有被写成当前状态。

## 四、API 配置持久化与恢复

API 配置属于独立的本地设置，不应因为清空练习记录、切换离线模式、重建题库或数据文件版本变化而丢失。

### 当前实现风险

当前 `apiKey` 与 `settings` 一起保存在 `shenlun_v2` 的 localStorage 中，同时整个 `D` 还可能从 File System Access API 连接的数据文件载入。`initData()` 载入数据文件时会用文件内容整体覆盖当前 `D`，因此旧数据文件如果没有最新的 `settings.apiKey`，可能把本地刚保存的 API 配置覆盖掉。这是“保存后重开又清空”的重点排查方向。

此外，直接双击单文件时，浏览器的 `file://` 存储行为可能因浏览器、路径或隐私设置而不稳定。不能只假设 localStorage 永远可靠。

### 目标行为

1. 填写 API Key，点击保存；
2. 关闭页面并重新打开；
3. 再进入设置时，提供商、Base URL、模型、API Key 都仍然存在；
4. 切换到离线模式后，API 配置仍保留，只是不调用；
5. 清空练习记录或重置题库后，API 配置仍保留；
6. 连接一个旧的数据文件时，不能用旧文件里的空 API 配置覆盖当前本地配置；
7. 数据文件保存后重新打开，API 配置仍能恢复；
8. API 请求失败不能自动清空输入框或覆盖已保存配置。

### 建议实现

把 API 配置从练习数据中拆成独立的设置存储，例如：

```js
const API_SETTINGS_KEY = 'shenlun_api_settings_v1';
// provider, apiKey, model, baseUrl, temperature, maxTokens
```

可以继续使用 localStorage 保存，也可以同时使用 IndexedDB 做冗余保存。重点不是换存储技术，而是做到：

- 练习数据和 API 配置分开读写；
- 旧数据文件缺少 API 配置时，保留本地已保存配置；
- `reset` 只清空练习数据，不清空 API 配置；
- 启动恢复时按明确优先级合并，而不是整个 `D` 直接覆盖；
- 不把真实 API Key 写进仓库、题库、Git 历史或日志；
- 设置界面保留密码框，必要时提供“显示/隐藏”按钮；
- 提醒使用者：本地保存是明文或可被本机浏览器读取的配置，不适合在共享电脑上保存长期 key。

### 必须验证的持久化场景

- 保存后刷新页面；
- 关闭浏览器后重新打开；
- 先连接旧数据文件，再打开设置；
- 切换离线，再切回在线；
- 清空练习历史；
- 仅修改模型或 Base URL 后重新打开；
- API 请求失败后重新打开；
- localStorage 中没有配置但旧数据文件中有配置；
- 两边配置冲突时，界面明确采用哪一份，不静默覆盖。

## 五、手机端使用与本地保存

手机端是实际使用场景，尤其要考虑上课期间用手机做几道题。目标不是让手机连接电脑上的文件，而是让手机浏览器能独立打开、练习、保存和恢复数据。

### 手机端目标

- Android 和 iPhone 浏览器都能打开练习器；
- 不依赖桌面端 `.bat`、Python 服务或本地开发环境；
- 手机窄屏下材料、事实选项、命名输入框和答案区不发生横向溢出；
- 触摸选择、输入法、滚动和提交按钮正常工作；
- 离线题库在没有网络时仍可使用；
- 练习记录自动保存在当前手机浏览器；
- 关闭页面后重新打开，题目记录、历史和设置仍能恢复。

### 本地保存边界

当前使用的 `localStorage` 和 IndexedDB 可以作为手机浏览器内的本地保存，但它不是跨设备同步，也不是用户可见的文件。数据会跟着浏览器、域名/来源和设备走，清理浏览器数据后可能丢失。

桌面端的 File System Access API 不能作为手机端唯一保存方案：手机浏览器支持情况不一致，iPhone 尤其不能按电脑方式持续写入用户选择的本地文件。因此手机端必须先保证浏览器本地存储独立可用。

### 必须增加的数据备份能力

为避免“手机清缓存后全没了”，在设置或记录页增加：

- `导出数据`：把练习记录、错题、画像、设置导出为 JSON 文件或分享文件；
- `导入数据`：从手机文件选择器恢复 JSON；
- `复制数据`：必要时提供复制为文本的兜底方式；
- 导入前显示数据时间和记录数量，并要求确认；
- 导入失败不能覆盖现有数据；
- API Key 默认不导出，或单独提供明确的“是否包含 API 配置”选项，并警告导出文件含有密钥。

### 手机端保存验收

必须实际测试：

1. 手机浏览器打开页面；
2. 完成一道离线题并产生历史记录；
3. 关闭页面后重新打开，历史仍然存在；
4. 刷新页面、切换横竖屏、切换离线/在线设置，数据不丢；
5. 导出 JSON，再清理测试数据，导入后记录恢复；
6. 导入错误或损坏文件时，原有数据不被覆盖；
7. Android 和 iPhone 至少各验证一次，不能只在电脑模拟手机；
8. 移动端 API 配置保存后重开仍存在，但不把 key 自动暴露在导出文件或页面可见文本中。

### 本轮部署决策：先做单文件下载版

本轮明确不部署 HTTPS，也不做 GitHub Pages、Cloudflare Pages、Netlify 或国内云平台发布。先把 `规范表达练习.html` 做成可以直接分享和下载的单文件版本。

使用方式：

- 电脑：下载后双击 HTML 打开；
- 手机：下载 HTML 后用浏览器或文件管理器打开；
- GitHub 私有仓库：只作为代码备份和版本管理，不作为主要访问入口；
- 练习数据：保存在当前设备的浏览器本地，并通过导出/导入 JSON 备份。

单文件本地打开时，不同手机对 `file://`、localStorage 和 IndexedDB 的支持可能不完全一致。因此执行 Agent 必须优先做好移动端兼容、自动保存和导入导出，不得因为“以后可能部署 HTTPS”而提前引入部署配置、登录系统或云端存储。

未来如果决定公开给别人使用，再单独讨论国内可稳定访问的 HTTPS 方案。这个选择不属于本轮执行范围。

## 六、评价方式：离线对照，AI 逐语义单元批改

### 1. 先区分“逐字逐句”

不建议按每个汉字机械扣分。申论答案允许同义替换、语序调整和不同但合理的规范表达，逐字比较会把正确答案误判为错误。

本计划中的“逐字逐句评价”具体指：

- 按句子或语义单元拆分用户答案；
- 每个语义单元对应一个材料事实、采分点或答案结构；
- 标出原文片段、对应的标准要点、判断结果和修改建议；
- 对规范词进行词语级提示，但不把单个字的差异直接当成扣分依据。

反馈应尽量回答：这一句说了什么、对应哪个得分点、少了什么、表达是否准确、下一步怎么改。

### 2. 离线模式：取消笼统评分，改为答案对照

离线模式不做一个看似精确但不可靠的总分。概括训练提交后，直接展示“你的答案 / 参考拆解 / 参考答案”的对照：

```text
你的答案：便民服务存在窗口材料要求不一、老人不会使用线上平台等问题。

对应情况：
✓ 服务标准不统一：已覆盖
✓ 数字服务适老性不足：已覆盖，但“老人不会使用”仍停留在事实层
✗ 线下服务承载力不足：未覆盖

参考拆解：
材料事实 → 共同含义 → 推荐概括词
```

离线反馈可以保留“覆盖 2/3 组要点”这类过程统计，但不要把它包装成正式考试分数，也不要显示“你得了 7.3 分”这类精确结论。

离线反馈的数据来源必须标注：

> 离线答案对照：根据本题预设采分点和可接受表达生成，不是 AI 批改，也不等同于真实考试得分。

规范改写模式的离线结果也采用“原句 / 你的改写 / 词库建议 / 参考表达”对照，并保留用户自评，不生成伪精确分数。

### 3. AI 模式：逐语义单元、逐句解释

AI 模式可以进行更开放的评价，但必须要求模型返回结构化结果，不接受一段笼统总评代替逐句批改。

建议返回结构：

```json
{
  "overall": "总体评价",
  "units": [
    {
      "userText": "用户答案中的原文片段",
      "matchedPoint": "对应的采分点或共同含义",
      "status": "covered|partial|missing|extra|uncertain",
      "problemType": "none|漏点|事实不准|归类不当|概括过窄|概括过宽|口语化|重复|逻辑不清",
      "comment": "为什么这样判断",
      "suggestion": "建议改法",
      "referenceText": "对应的参考表达",
      "score": 0
    }
  ],
  "dimensions": {
    "要点覆盖": 0,
    "概括准确": 0,
    "表达规范": 0,
    "结构完整": 0
  },
  "nextFocus": "下一次练习重点"
}
```

AI 批改流程：

1. 先把参考答案拆成采分点和语义单元；
2. 再把用户答案按标点和语义切分；
3. 将每个用户单元匹配到一个或多个采分点；
4. 标记覆盖、部分覆盖、漏点、误归类、冗余或无法判断；
5. 对每个问题给出原文片段、原因、建议表达和参考表达；
6. 最后再给维度分数和总体评价。

如果模型无法可靠判断某个表达是否等价，应标记为 `uncertain`，提示用户对照，而不是直接扣分。

### 4. 逐句反馈的界面要求

不需要新增复杂页面，可以在现有结果区里增加按语义单元排列的反馈行：

```text
你的表达：老人不会操作
对应要点：数字服务适老性不足
判断：部分覆盖
说明：这是材料事实，还没有概括出背后的服务问题
建议：可概括为“数字服务适老性不足”
```

离线模式显示题库预设的确定性对照。AI 模式显示模型生成的逐句反馈，并在顶部明确标注来源。两者不能共用一个没有来源说明的“综合评分”。

## 六、本轮明确不做的事

本轮不要：

- 把“逐字逐句”实现成每个汉字机械比对；
- 在离线模式中用规则生成伪精确总分；
- 用一段总体评语代替 AI 的逐语义单元反馈；
- 重做整体视觉风格；
- 更换单文件 HTML 架构；
- 新增登录、排行榜、分享、账号系统；
- 新增复杂的 AI 对话页面；
- 把所有题目一次性扩展成真正完整的整套申论材料；
- 为了“看起来更丰富”继续堆词库、卡片和动画；
- 改动 File System Access API、localStorage、IndexedDB 等已有数据持久化机制。

UI 只允许做必要的小调整，继续沿用当前“纸面答题卡”布局和三步结构。

## 六、当前主要问题

### 1. 材料与训练目标不匹配

当前题目已经有 `material`、`sentences`、`points`、`ref`，但材料整体偏短，且使用者更像是在选择预先拆好的句子，而不是从完整材料中主动找事实。

问题不在于每道题必须达到正式申论大题的完整篇幅。本轮先把材料控制在约 150 至 350 个汉字，作为微型材料训练。后续如果机制验证有效，再逐步增加至 400 至 800 字或更长。

### 2. 评分逻辑过度依赖字符串包含

离线规则可以用字符串匹配辅助定位，但不能把字符串命中等同于真实评分。AI 模式需要单独进行语义单元匹配，并把不确定判断显式标记出来。

当前 `assessSummary()` 主要通过 `includes()` 判断：

- 是否选中关键句；
- 是否覆盖某个 point；
- 用户是否输入了参考词的片段；
- 是否在答案里出现某些关键词。

这会产生两个问题：

- 使用者可能只是碰巧写中了关键词，但没有真正完成归类；
- 使用者用了合理的同义表达，却因为没有出现参考词而被判低分。

### 3. 解析告诉了结果，没有解释过程

当前反馈已经有“找点、归类、命名、表达”四层，但展示内容偏向数量统计和参考答案，缺少：

- 漏掉哪条事实；
- 误选了哪条事实；
- 哪些事实构成同一组；
- 当前答案和参考归类之间的差异；
- 为什么某个概括词比口语词更合适。

本轮要把反馈从“判分”改成“指出下一次怎么做”。

## 七、题库数据结构调整

保留现有字段，给 `SUMMARY_BANK` 的题目增加以下字段。不要删除旧字段，避免历史记录和旧数据报错。

```js

  material: "完整材料……",
```

### 数据要求

第一轮只需要重做或新增 6 道高质量样板题，不要急着一次性改完全部题库。建议覆盖：

- 问题：2 道
- 做法：1 道
- 成效：1 道
- 变化：1 道
- 经验：1 道

每道材料约 150 至 350 字，包含 3 至 5 组核心含义，至少加入 1 条不应被概括为核心要点的背景或例子。材料要像调研材料，不要写成词语练习题。

每个核心 point 必须至少对应 1 条事实，重要 point 最好对应 2 条相互支撑的事实。不要让 `name` 直接复述材料原词，例如材料写“跑几个窗口”，概括词应当是“办事流程繁琐”或“服务流程不够优化”，而不是“跑窗口问题”。

## 八、交互调整方案

不重做页面布局，继续保留：

- 01 圈点
- 02 命名
- 03 成句

但调整每一步的含义和文案。

### 01 圈点

当前是“选出有效分句”。改成：

> 圈出能支撑答案的事实，注意排除背景、例子和修辞性描述。

如果当前 UI 仍然展示拆开的句子，可以先保留句子选择交互，不必马上做真正的文本划词。为了不扩大 UI 改造范围，本轮把 `sentences` 视为材料中的“事实片段”，但材料本身必须完整连贯。

提交后明确区分：

- 漏选事实；
- 误选背景或干扰项；
- 选中了事实但没有覆盖对应 point。

### 02 命名

当前允许用户用顿号输入一串名称。保留该输入方式，但反馈时需要把用户输入拆成名称数组，逐个和标准 point 比较。

不要只做简单的 `names.includes(point.name)`。至少支持：

- 完全匹配；
- 去除“问题、现象、不足、程度”等常见尾词后的匹配；
- 预先定义的可接受同义表达；
- 无法判断时标记为“需人工复核”，不要直接判错。

每个 point 增加 `acceptableNames` 可选字段，例如：

```js
acceptableNames: ["服务标准不统一", "服务标准不够统一", "窗口服务标准不一"]
```

反馈文案要说明差异：

> 你写的是“老人不会操作”，这是材料事实，不是上位概括。可以概括为“数字服务适老性不足”。

### 03 成句

继续保留一个答案输入框和字数统计。重点不是增加新的输入框，而是让提交后展示答案的结构拆解：

```text
总括对象：便民服务
问题1：服务标准不统一
问题2：数字服务适老性不足
结尾：等问题
```

如果答案遗漏某一组 point，要明确显示：

> 你的答案已经写到“服务标准不统一”，但没有覆盖“线下服务承载力不足”。

## 九、反馈模块重做

保留现有四层反馈视觉，但重写每层的内容。每层都要包含“结果 + 证据 + 下一步建议”。

### 1. 找点反馈

显示：

- 核心事实覆盖数；
- 漏选事实列表；
- 误选事实列表；
- 如果有干扰项，说明为什么不应纳入答案。

示例：

```text
找点：覆盖 4/5 条有效事实
漏选：高峰时段排队时间较长
误选：大厅位于一楼
原因：这是地点背景，不能单独形成问题要点
```

### 2. 归类反馈

显示标准归类表，至少包含：

| 事实 | 应归入的共同含义 | 你的状态 |
| --- | --- | --- |
| 材料要求不一致、反复补交 | 服务标准不统一 | 已覆盖 |
| 入口较多、老人不会操作 | 数字服务适老性不足 | 漏掉 |

如果当前 UI 没有让用户单独输入归类关系，可以根据“选中的事实 + 用户命名”推断，并明确标注：

> 当前版本只检查你是否覆盖了这组事实，没有要求你逐组拖拽归类。

不要假装用户已经完成了系统没有提供的归类操作。

### 3. 命名反馈

逐个检查 point：

- 已准确概括；
- 仍停留在材料事实层面；
- 过于宽泛；
- 与题型不匹配；
- 使用口语词；
- 暂时无法判断。

示例：

```text
“老人不会操作”是具体事实，不是概括词。
它反映的共同问题是“数字服务适老性不足”。
```

### 4. 表达反馈

检查：

- 是否覆盖全部核心 point；
- 是否先总括对象，再列举问题；
- 是否出现事实重复；
- 是否把原因、做法、成效混成同一层；
- 是否出现明显口语表达；
- 字数是否明显超限或过短。

不要用简单关键词出现与否代替完整评价。关键词匹配只能作为辅助信号。

## 十一、评分规则

离线模式不输出正式总分，只输出答案对照和过程统计，例如“覆盖 2/3 组采分点”。这个统计不是考试得分。

AI 模式可以保留 10 分制，但分数只作为逐句批改后的摘要，不得替代逐句反馈。AI 分数必须来自独立的维度评分和结构化 `units`，不能复用离线结果冒充 AI 评分。

本节规则只针对离线模式和 AI 模式的来源隔离：离线不调用模型、不做开放式语义判定；AI 批改要标明模型来源，并允许 `uncertain`，避免虚假的精确性。

建议权重：

- 找点 3 分
- 归类 2 分
- 命名 3 分
- 成句 2 分

评分必须基于结构化结果：

```js
{
  selectedFacts,
  missedFacts,
  distractors,
  coveredPoints,
  missedPoints,
  namesMatched,
  namesPartial,
  namesUncertain,
  expressionIssues,
  score
}
```

避免以下做法：

- 只检查答案是否包含参考词；
- 用参考词前两个字、后两个字直接判定正确；
- 用户写了合理同义词却直接判错；
- 把“显示参考答案”当成解析完成。

如果规则无法可靠判断，显示“部分匹配”或“需要对照判断”，不要伪装成精确评分。

## 十一、实施顺序

### 阶段 1：先做一题样板

1. 选择现有 `SUMMARY_BANK` 中的一道 L1 问题题；
2. 将材料改成 150 至 350 字的连续微型材料；
3. 为材料补充真实来源或明确标记为原创综合材料，并核对现实时间线；
4. 补齐事实片段、point 映射、干扰项和解析说明；
4. 改造 `assessSummary()` 返回结构化结果；
5. 改造提交后的反馈区域；
6. 手动完成这道题，检查反馈是否能解释“为什么错”。

阶段 1 的验收标准：做错时，使用者不看参考答案也能知道自己漏了什么、误选了什么、概括词为什么不准确。

### 阶段 2：扩展 6 道样板题

按照问题、做法、成效、变化、经验五类补齐 6 道题。每类题的反馈文案不能都套用“问题覆盖”，要根据题型变化：

- 做法：是否识别出动作、对象、方式和机制；
- 成效：是否识别出结果、变化和受益对象；
- 变化：是否同时抓住变化前后和变化方向；
- 经验：是否能从多个做法中抽出可迁移原则。

### 阶段 3：回归与验证

完成以下检查：

- 单文件可直接打开；
- 首次启动不再卡在“铺纸研墨”；
- 概括训练和规范改写都能正常切换；
- 换题不会残留上一题的事实选择、命名和答案；
- 提交后可以重新作答；
- 下一题可以正常进入；
- 历史记录仍能保存；
- 旧历史记录没有因为新增字段报错；
- 暗色模式、移动端宽度、键盘输入没有明显错位；
- `node --check` 通过；
- 至少手动验证 1 道问题题、1 道做法题、1 道成效题。

## 十二、给执行 Agent 的工作边界

执行前先阅读：

- `CLAUDE.md`
- 当前 `规范表达练习.html`
- 相关 `SUMMARY_BANK` 数据和 `assessSummary()`、`submitSummary()`、反馈渲染部分

执行规则：

- 先分别检查离线模式和 AI 模式的入口、状态显示与降级路径；
- 先检查 API 配置保存、恢复、重置和旧数据文件覆盖逻辑；
- 只改当前项目目录里的单文件 HTML 和必要的备份；
- 改之前在 `_backups/` 保存一份版本；
- 不新增框架、不引入构建工具、不拆分外部依赖；
- 不重做整体视觉；
- 不把所有历史题目一次性机械改写；
- 先完成 1 道样板题并验证机制，再扩展到 6 道；
- 如果发现当前 UI 无法真实表达某个反馈，不要编造用户已经完成的操作，改为显示“根据已选事实推断”的说明；
- 离线反馈只能使用题库映射和确定性规则，不能偷偷调用 AI；
- 离线概括训练不输出伪精确总分，改为答案对照和采分点覆盖情况；
- AI 反馈必须标注来源，不能把离线规则结果写成 AI 结论；
- AI 批改必须返回逐语义单元结果，不能只返回一段笼统评价；
- 完成后报告：改动文件、题目数量、每道样板题的来源与时效审核结果、离线/AI 边界、API 配置持久化测试结果和未解决风险。

## 十三、最终判断标准

这轮优化成功，不是因为页面更复杂，也不是因为反馈卡片更多，而是因为离线模式和 AI 模式各自做清楚了自己的事，并且离线模式本身已经能让使用者完成一次真实的认知练习：

> 从一段材料中找出分散事实，判断哪些事实属于同一个问题或做法，给这组事实命名，再把它组织成申论答案。

如果使用者做完后只是看到了一个分数和一份参考答案，说明这轮没有完成。
